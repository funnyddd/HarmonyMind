## 三、所有可能被问到的面试问题 + 答案

### 🏗️ 架构设计类

#### Q1: 项目的整体架构是怎样的？为什么选择这个架构？

**答案：** 我采用了 **Clean Architecture（整洁架构）** 分层设计：

- **Data 层**：`NoteRepository` + `LocalNoteDataSource`，封装 RelationalStore 数据库操作，返回统一的 `Result<T>` 类型
- **Domain 层**：UseCase（`CreateNoteUseCase` / `GetNotesUseCase` / `DeleteNoteUseCase` / `SearchNotesUseCase`），每个用例只做一件事，符合单一职责原则
- **Presentation 层**：MVVM 模式，ViewModel 持有状态和业务逻辑，Page/Component 只负责 UI 渲染

**为什么选 Clean Architecture：**
1. **可测试性**：每一层可以独立 mock 测试（项目已集成 Hypium + Hamock）
2. **可扩展性**：后续从本地存储切换到"本地 + 分布式"双数据源时，只需改 Repository，Domain 和 UI 层不受影响
3. **官方推荐 MVVM，Clean Architecture 是其自然延伸**：HarmonyOS 官方文档明确推荐 MVVM 架构（View → ViewModel → Model），并提供了 `@ObservedV2`/`@Trace`/`@ComponentV2` 等状态管理装饰器。我在官方 MVVM 的基础上进一步拆出独立的 Domain 层（UseCase）和 Data 层（Repository → DataSource），形成更完整的 Clean Architecture，便于后续扩展分布式和 AI 功能

#### Q3: ServiceFactory 是做什么的？用了什么设计模式？

**答案：** `ServiceFactory` 使用了两个设计模式：

1. **单例模式（Singleton）**：`getInstance()` 确保全局只有一个工厂实例
2. **工厂模式（Factory）+ 懒加载（Lazy Initialization）**：每个服务/用例第一次访问时才创建，之后缓存复用

```typescript
export class ServiceFactory {
  private static instance: ServiceFactory;
  private noteRepository: NoteRepository | null = null;  // 懒加载
  
  static getInstance(): ServiceFactory { ... }           // 单例
  
  getNoteRepository(): NoteRepository {
    if (this.noteRepository === null) {
      this.noteRepository = new NoteRepository();        // 延迟初始化
    }
    return this.noteRepository;
  }
}
```

**好处：** 避免在页面中到处 new 对象，集中管理依赖；后续改为分布式数据源只需改工厂内部，调用方无感知。

---

#### Q4: `Result<T>` 类型是干什么的？为什么不直接用 try-catch？

**答案：** `Result<T>` 是一个统一的业务结果封装，借鉴了 Rust/Swift 的 Result 类型。

```typescript
export class Result<T> {
  readonly isSuccess: boolean;
  readonly data: T | null;
  readonly error: string;
  readonly code: number;
  
  static success<T>(data: T): Result<T> { ... }
  static failure<T>(error: string, code?: number): Result<T> { ... }
}
```

**为什么不用 try-catch：**
1. **类型安全**：调用方必须显式检查 `isSuccess` 才能访问 `data`，编译器无法跳过
2. **ArkTS 限制**：ArkTS 严格模式不支持 discriminated union 类型收窄，所以用单 class + boolean 标志位替代
3. **统一错误处理**：所有层（DataSource → Repository → UseCase）返回同一类型，上层统一用 `fold()` / `getOrDefault()` 处理

### 🗄️ 数据存储类

#### Q5: 数据库是怎么设计的？有哪些表和索引？

**答案：** 使用 HarmonyOS **RelationalStore（关系型数据库）**，两张表：

**notes 表：**
```sql
CREATE TABLE notes (
  id TEXT PRIMARY KEY,
  title, content, preview,
  category_id, category_name, category_color, category_icon,
  created_at, updated_at,
  ai_summary, has_ai_summary,
  sync_status, is_favorite, source_device,
  version INTEGER  -- 乐观锁版本号，用于冲突检测
)
```

**categories 表：**
```sql
CREATE TABLE categories (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL UNIQUE,
  color, icon_name, created_at, sort_order
)
```

**三个索引：**
- `idx_notes_updated_at` → 按时间排序查询
- `idx_notes_category` → 按分类筛选
- `idx_notes_sync_status` → 同步状态查询

---

#### Q6: version 字段是用来干什么的？

**答案：** 这是**乐观锁（Optimistic Locking）**的版本号。

- 每次更新笔记时 `version += 1`
- 用于分布式同步场景的**冲突检测**：当两台设备同时修改同一条笔记，同步时对比 version，高版本覆盖低版本，避免数据丢失
- 这是为 Phase 3 分布式同步预留的设计

---

#### Q7: 批量插入用了事务吗？为什么？

**答案：** 用了。在 `insertNotes()` 中使用 `store.beginTransaction()` / `store.commit()`：

```typescript
async insertNotes(notes: Note[]): Promise<Result<boolean>> {
  store.beginTransaction();
  for (let i = 0; i < notes.length; i++) {
    await store.insert(DB_TABLE_NOTES, bucket);
  }
  store.commit();
}
```

**为什么：**
1. **性能**：批量操作在事务中远快于逐条提交（减少 fsync 次数）
2. **原子性**：分布式同步接收到的批量笔记要么全部写入，要么全部回滚，不会出现部分写入的脏状态

### 🔍 搜索与性能类

#### Q8: 搜索功能是怎么实现的？有什么优化？

**答案：** 实现了以下优化：

1. **SQL LIKE 模糊搜索**：同时对 `title` 和 `content` 字段做 `LIKE %keyword%`
2. **防抖（Debounce）**：用户输入后 300ms 才触发搜索，避免频繁查询
3. **搜索历史**：本地缓存最近 10 条搜索关键词，去重 + LRU 队列
4. **LIMIT 50**：搜索结果限制 50 条，避免返回过多数据

```typescript
// 300ms 防抖
private onSearchTextChanged(value: string): void {
  clearTimeout(this.searchTimeout);
  if (value.trim().length > 0) {
    this.searchTimeout = setTimeout(() => {
      this.performSearch(value);
    }, 300);
  }
}
```

---

#### Q9: 如果笔记量很大（比如10万条），你会怎么优化？

**答案：**

1. **数据库层面**：加全文索引（FTS），替换 LIKE 模糊搜索
2. **分页加载**：当前已支持 `limit` + `offset`，改为游标分页更高效
3. **列表虚拟化**：`List` 组件本身支持懒加载，只渲染可见区域的 Item
4. **预览字段冗余**：`preview` 字段提前截取前100字，列表页不需要读完整 `content`

### 🌐 分布式与同步类（Phase 3 规划）

#### Q10: 分布式同步是怎么设计的？

**答案：**

1. **KVStore**：使用 HarmonyOS `distributedKVStore`，以 `note_${id}` 为 key 存储笔记
2. **同步状态机**：每条笔记有 `syncStatus` 字段 — `local_only` → `pending` → `synced` / `conflict`
3. **乐观锁**：`version` 字段做冲突检测
4. **权限**：`ohos.permission.DISTRIBUTED_DATASYNC`

#### Q11: Continuation（跨设备迁移）是怎么实现的？

Continuation 是 HarmonyOS 的核心分布式能力——用户在手机上编辑笔记，靠近平板后编辑状态无缝迁移。

**流程：**
1. **设备A `onContinue()`**：序列化当前编辑状态（noteId、光标位置、滚动偏移）→ 通过分布式软总线传递
2. **设备B `onRestore()`**：解析 Continuation 数据 → 恢复笔记 ID、光标位置、滚动位置
3. **`onNewWant()`**：检测 `LaunchReason.CONTINUATION` 触发恢复

```typescript
onContinue(wantParam: Record<string, Object>): AbilityConstant.OnContinueResult {
  wantParam['continueData'] = JSON.stringify({ noteId, cursorPos, scrollOffset });
  return AbilityConstant.OnContinueResult.AGREE;
}
```

### 🤖 AI 功能类（Phase 4 规划）

#### Q12: AI 摘要是怎么实现的？

1. 获取笔记内容 → 调用 `AIService.summarize(note)` → 得到 `AISummaryResult`（含摘要文本、关键词标签、情感分析、建议分类）
2. 支持双来源：**云端**（API 调用，有 token 统计）和**端侧**（本地模型推理）
3. **批量处理**：每批 3 条并发，`Promise.all` 并行请求
4. 结果持久化到 `ai_summary` 字段

---

#### Q13: AI 分类是怎么实现的？规则引擎和 AI 怎么配合？

采用 **AI + 规则引擎双通道 + 降级策略**：

1. **AI 分类（可选）**：调用 `AIService.categorize()`，返回分类名 + 置信度
2. **规则引擎（兜底）**：纯本地关键词匹配，预定义了 7 类关键词库（工作/学习/生活/灵感/会议/读书/旅行）
3. 如果 AI 失败 → 自动 fallback 到规则引擎

```typescript
async execute(noteId: string, useAI: boolean = false): Promise<Result<NoteCategory>> {
  if (useAI && this.aiService !== null) {
    return await this.categorizeWithAI(note);    // AI 优先
  }
  return await this.categorizeByRules(note);     // 规则兜底
}
```

### 🧩 ArkTS/鸿蒙特性类

#### Q14: ArkTS 严格模式有哪些限制？你是怎么处理的？

**答案：** ArkTS 严格模式限制了 TypeScript 的部分动态特性：

1. **不支持 discriminated union 类型收窄** → 改用 `Result<T>` 单 class + `isSuccess` boolean
2. **不允许 `any` / `unknown` 随意使用** → 所有类型显式声明，`ValuesBucket` 转换通过中间 class + `as Object as` 双重断言
3. **严格的大小写检查** → `build-profile.json5` 中启用了 `caseSensitiveCheck: true`
4. **规范化的 OHM URL** → 启用 `useNormalizedOHMUrl: true`

#### Q15: @Observed 和 @ObjectLink 是怎么用的？和 @State 有什么区别？

| 装饰器 | 作用 | 使用场景 |
|--------|------|---------|
| `@State` | 组件内部状态，变化触发 re-render | Page 持有 ViewModel |
| `@Observed` | 标记 class 为可观察 | Model / ViewModel 类 |
| `@ObjectLink` | 引用父组件的 @Observed 对象 | 子组件接收 Model 实例 |

我的使用：
- `Note`、`NoteCategory` 用 `@Observed` 标记
- `NoteCard` 中用 `@ObjectLink note: Note` 接收，修改 note 属性自动刷新 UI
- Page 中用 `@State` 持有 ViewModel

### 📐 设计模式与工程实践类

#### Q18: 你在这个项目里用了哪些设计模式？

1. **Singleton（单例）**：`ServiceFactory`、`DatabaseConfig.store`
2. **Factory Method（工厂）**：`ServiceFactory.getXxxUseCase()`
3. **Repository Pattern（仓库模式）**：`NoteRepository` 抽象数据源
4. **Strategy（策略模式）**：AI 分类 vs 规则引擎分类，运行时选择（`CategorizeNoteUseCase`）
5. **Result Pattern**：`Result<T>` 统一错误处理
6. **Observer（观察者）**：`@Observed` + `@ObjectLink` 响应式数据流
7. **Facade（外观）**：UseCase 封装复杂子系统的简单接口

#### Q19: 你是怎么处理错误的？

统一的三层错误处理：

1. **数据层**：catch 所有数据库异常 → 包装成 `Result.failure(ERROR_DB_OPERATION, code, msg)`
2. **领域层**：UseCase 检查前置条件（如笔记是否存在）→ 返回业务错误 `Result.failure(ERROR_NOTE_NOT_FOUND, 404)`
3. **表现层**：ViewModel 检查 `result.isSuccess` → 设置 `errorMessage` → UI 展示 Toast

```typescript
// Page 层统一处理
const saved = await this.viewModel.save();
if (saved) {
  promptAction.showToast({ message: '已保存' });
} else {
  promptAction.showToast({ message: this.viewModel.errorMessage });
}
```

#### Q21: 脏检查（Dirty Check）是怎么实现的？

`NoteEditorViewModel` 保存时对比当前值和原始值：

```typescript
private checkDirty(): void {
  this.isDirty =
    this.note.title !== this.originalNote.title ||
    this.note.content !== this.originalNote.content ||
    this.note.category?.name !== this.originalNote.category?.name;
}
```

- 新建笔记时 `originalNote` 是空 Note
- 编辑已有笔记时 `originalNote` 是加载时的深拷贝（`note.clone()`）
- 保存成功后 `originalNote` 更新为当前值的深拷贝
- 返回按钮自动检测 `isDirty`，有修改则先保存再返回

---

#### Q22: 数据模型为什么有 `clone()` 方法？

1. **脏检查需要原始副本**：ViewModel 中 `originalNote` 是加载时的深拷贝
2. **不可变数据模式**：避免直接修改引用导致意外的副作用
3. **ArkTS 限制**：ArkTS 没有 JavaScript 的 `structuredClone` 或 spread 展开的完全自由，所以手写 `clone()`

### 🟡 2分钟标准版（一面/二面）

> **「项目背景」**
> HarmonyMind 是我独立开发的 HarmonyOS 原生笔记应用。选这个方向是因为 HarmonyOS 生态正在快速增长，我想深入理解 Stage 模型、ArkTS 语言和分布式能力。
>
> **「技术架构」**
> 架构上我选择了 **Clean Architecture 三层分离**：
> - **Data 层**用 Repository 模式封装了 RelationalStore，支持事务批量写入和乐观锁版本控制
> - **Domain 层**拆成了多个 UseCase，每个只做一件事，通过 ServiceFactory 单例工厂做依赖注入
> - **Presentation 层**用 MVVM，ArkUI 声明式 UI，@Observed/@ObjectLink 做响应式数据绑定
> - 所有层之间用统一的 `Result<T>` 类型传递结果，避免异常泛滥
>
> **「核心功能」**
> - 笔记 CRUD + 8 种预设分类 + 全文模糊搜索（300ms 防抖 + 搜索历史）
> - 下拉刷新、长按删除、脏检查自动保存等体验细节
> - 设计了分布式同步架构：syncStatus 状态机 + 乐观锁版本号 + KVStore
> - 设计了跨设备 Continuation：编辑中手机靠近平板，笔记状态无缝迁移

### 🔴 5分钟深度版（终面/技术面）

在 2 分钟版基础上追加以下内容：

> **「为什么这样设计——我的思考」**
>
> **1. 为什么用 Clean Architecture 而不是简单 MVC？**
> 因为这个项目规划了三大阶段：本地 → 分布式 → AI。如果一开始不做好分层，后面加分布式数据源或 AI 服务时就会到处改代码。现在只需在 Repository 层扩展，Domain 和 UI 层不用动。
>
> **2. 为什么手写 Result<T> 而不是用 try-catch？**
> ArkTS 严格模式不支持 discriminated union，所以我参考了 Rust 的 Result 和 Swift 的 Result，用单 class + boolean 实现。这样每一层返回类型都是统一的，上层必须显式判断 isSuccess，写出来的代码更安全。
>
> **3. 分布式同步怎么保证数据一致性？**
> 我用的是"乐观锁 + 状态机"方案：每条笔记有 version 字段，每次更新+1；syncStatus 字段标记 local_only/pending/synced/conflict。同步时对比 version，高版本覆盖低版本。这不是最强的方案（强一致需要 CRDT），但对笔记场景已足够，且实现成本低。
>
> **4. AI 分类为什么加规则引擎兜底？**
> 因为 AI 服务可能不可用（无网络、API 限流、超时 30s），用户体验不能依赖 AI。所以默认走规则引擎，AI 是可选增强。这是"Graceful Degradation"的实践。
>
> **「不足和改进方向」**
> - 目前主要是本地单机版，分布式同步和 AI 功能在备份代码中有设计但还没完整联调
> - 如果笔记量很大（10万+），需要用 FTS 全文索引替代 LIKE 模糊搜索
> - 可以用游标分页替代当前简单的 limit/offset 分页
> - 单元测试覆盖率还需要提升
